# Timeline Playbook

Use this reference only after the Requirements Receipt is confirmed. Turn transcript beats into varied but coherent treatments, adapting every timing suggestion to the confirmed aspect, style, intensity, and scope.

## Information Arc

| Spoken job | Preferred treatment |
|---|---|
| Strong claim | Kinetic two-hit title with one proof signal |
| Promise or scope | Short editorial title assembly |
| Tool pairing | Split evidence, connector, or source-to-result chain |
| Prepare source | Format/spec information strip or source card |
| Install/setup | Screenshot context, detail crop, readiness state |
| Parameter explanation | Large parameter demo, phone frame, gauge, or option selector |
| Transcript timing | Speech-to-keyword timeline with playhead or mapped markers |
| Plan review | Checklist, matrix, or safe-area comparison |
| Build/generate | Code-to-timeline-to-preview pipeline |
| Preview | Full-screen evidence plus three concrete quality checks |
| Natural-language revision | Before/after emphasis, duration, and position demos |
| Export | Focused action, progress, and receipt state |
| Summary | Connected process flow with spoken-step timing |
| Final thesis | Simplify the frame and land one memorable principle |

## Pacing Rules

- Align changes to semantic speech boundaries, not arbitrary equal intervals.
- In dense short-form teaching, introduce a meaningful new visual state every `4-8s`.
- Use a near-full takeover every `10-18s` only when real evidence supports it.
- Within long explanations, reveal sub-points on their exact spoken times instead of showing everything at scene start.
- Hold the final readable state long enough for a phone viewer to understand it.
- Prefer a hard cut when the spoken idea changes sharply; use a motion handoff when two ideas are causally connected.

## Opening Blueprint

For the first `15-20s`, prefer this shape:

1. `0-4s`: strongest result or claim, with a fast reveal and large keywords.
2. `4-7s`: promise what the viewer will understand.
3. `7-11s`: show the real tools, interface, or evidence.
4. `11-15s`: show source/spec preparation.
5. `15-20s`: begin the first concrete action and cut on a complete phrase.

Do not force these exact seconds when the speech differs. Preserve complete phrases and move the boundaries to the transcript.

## Screenshot Mapping

Create an asset map with:

| Field | Meaning |
|---|---|
| `spoken_start/end` | Exact line interval |
| `asset` | Local source path |
| `semantic_target` | What the viewer should notice |
| `crop` | Full view, region, or derived closeup |
| `callout` | Short phone-readable explanation |
| `face_coverage` | None, partial, or takeover |
| `subtitle_clear` | Required true |

If the asset does not prove the current line, postpone or omit it.

## Variety Without Noise

Across a full tutorial, rotate visual jobs rather than merely changing colors:

- Claim typography
- Screenshot proof
- Diagram or relationship
- Parameter or comparison demo
- Timeline or progress visualization
- Checklist or verdict
- Before/after revision
- Process summary

Do not place UI cards inside UI cards. Do not repeat the same screenshot zoom, whoosh, or chapter badge for every section.

## Review Times

Always inspect:

- `0.4s` and the first claim landing
- Every major scene start
- Midpoint of each screenshot takeover
- Every dense callout hold
- Every SFX cue
- The final thesis landing and final frame

At each time, verify phone readability, source subtitle safety, intentional face coverage, asset clarity, and deterministic seek behavior.
